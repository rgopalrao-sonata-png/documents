# Highlight Update API Review

## Frontend API handoff

Use this section as the implementation handoff for frontend wiring.

### Base URL

- Local admin API base: `http://localhost:18160/api/v1/highlight-sets-admin`

### Authentication

- Required on all admin highlight endpoints
- Header format:

```http
Authorization: JWT <token>
Content-Type: application/json
```

### Authorization requirement

- User must have the `enterprise_catalog_admin` role for the target enterprise.

### Local seeded values

- Enterprise UUID: `aaaaaaaa-0000-0000-0000-000000000001`
- Enterprise name: `Test Multi-License Enterprise`
- Enterprise slug: `test-multi-enterprise`
- Highlight set UUID: `cccccccc-0000-0000-0000-000000000001`
- Example highlighted content UUIDs:
   - `56cca5e9-9547-417b-a4e9-6b98b4b26693` → `course-v1:edX+CS101+2023`
   - `5a0c4988-4f04-444a-b92f-37f85d5e286f` → `course-v1:edX+PY101+2023`
   - `a449ed23-f463-4b71-af1d-dd8a8d893cb9` → `course-v1:edX+DemoX+Demo_Course`

### Endpoint summary

| Purpose | Method | Path |
|---|---|---|
| List highlight sets for an enterprise | `GET` | `/api/v1/highlight-sets-admin/?enterprise_uuid=<enterprise_uuid>` |
| Rename a highlight only | `POST` | `/api/v1/highlight-sets-admin/<highlight_set_uuid>/edit-highlight-title/` |
| Update title and/or courses in one request | `PATCH` | `/api/v1/highlight-sets-admin/<highlight_set_uuid>/edit-highlight/` |
| Add courses only | `POST` | `/api/v1/highlight-sets-admin/<highlight_set_uuid>/add-content/` |
| Remove courses only | `POST` | `/api/v1/highlight-sets-admin/<highlight_set_uuid>/remove-content/` |
| Favorite / unfavorite one highlighted item | `POST` | `/api/v1/highlight-sets-admin/<highlight_set_uuid>/toggle-favorite-highlight/` |

- Empty `title` → `400`

### 3. Combined edit endpoint

Use this when the frontend wants to update title and course membership in a single request.

#### Request

```http
PATCH /api/v1/highlight-sets-admin/cccccccc-0000-0000-0000-000000000001/edit-highlight/
Authorization: JWT <token>
Content-Type: application/json

{
   "title": "Bruno Rename Test",
   "add_content_keys": ["course-v1:edX+CS101+2023"],
   "remove_content_keys": ["course-v1:edX+PY101+2023"]
}
```

#### Success response

```json
{
   "highlight_set": {
      "uuid": "cccccccc-0000-0000-0000-000000000001",
      "title": "Bruno Rename Test",
      "is_published": true,
      "enterprise_curation": "bbbbbbbb-0000-0000-0000-000000000001",
      "card_image_url": "https://placekitten.com/400/225",
      "highlighted_content": [
         {
            "uuid": "56cca5e9-9547-417b-a4e9-6b98b4b26693",
            "content_key": "course-v1:edX+CS101+2023"
         }
      ]
   },
   "added_content_keys": ["course-v1:edX+CS101+2023"],
   "removed_content_keys": ["course-v1:edX+PY101+2023"],
   "ignored_content_keys": []
}
```

#### Validation and behavior

- At least one of `title`, `add_content_keys`, or `remove_content_keys` must be present.
- Invalid add keys are not rejected; they are returned in `ignored_content_keys`.
- Remove requests for missing keys are effectively ignored.
- If adding content exceeds the per-highlight backend limit, response is `403`.
- Although the docstring says “atomic”, current implementation is not wrapped in `transaction.atomic()`.

### 6. Toggle favorite

#### Request

```http
POST /api/v1/highlight-sets-admin/cccccccc-0000-0000-0000-000000000001/toggle-favorite-highlight/
Authorization: JWT <token>
Content-Type: application/json

{
   "content_uuid": "56cca5e9-9547-417b-a4e9-6b98b4b26693",
   "favorite": true

```

#### Success response

```json
{}
```

#### Notes

- Success status is `201` with empty JSON body.
- `content_uuid` must be the highlighted content row UUID, not the course key.
- `favorite` must be a boolean-like value accepted by backend parsing.

### Common error format

Most validation failures return this shape:

```json
{
   "Error": "human-readable message"
}
```

### Recommended frontend usage

1. Load page with the list endpoint filtered by `enterprise_uuid`.
2. Use `edit-highlight/` for edit modal save when title and course changes can happen together.
3. Use `edit-highlight-title/` only if the UI performs title-only save.
4. Use `highlighted_content[].uuid` for favorite toggles.
5. Treat `ignored_content_keys` as non-fatal warnings and show them in the UI if needed.

## Executive Summary

### Overall verdict

The backend implementation is **partially satisfied**.

### What is implemented

- Authenticated admin endpoint for renaming a highlight
- Authenticated admin endpoint for updating title + adding/removing associated course keys
- Role-based access control via JWT/session auth + `edx-rbac`
- Validation for empty / too-long titles
- Validation for missing payload
- Validation for highlight-set content count limit
- Logging for title/content updates
- Automated tests for success, unauthenticated, unauthorized, not-found, and limit failures
- Responses are rich enough for frontend state updates without a follow-up fetch

### Gaps / caveats

- The update payload uses **`content_keys`** / `add_content_keys` / `remove_content_keys`, not course UUIDs/IDs
- Invalid add-course keys are **ignored**, not rejected with a 4xx validation error
- There is **no full replacement** endpoint for the whole course list; only add/remove semantics exist
- No uniqueness validation for title within an enterprise
- Error responses are simple `{"Error": "..."}` strings, not field-level structured validation payloads
- The combined `PATCH` endpoint claims atomic behavior in the docstring, but the implementation is **not wrapped in a DB transaction**
